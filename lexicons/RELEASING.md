# Publishing lexicon releases

Lexicon packs are versioned independently from the MIT application. Release
metadata lives in [`release.toml`](release.toml); V1 uses the immutable Git tag
`lexicons-v1.0.0`, while the application remains at its own version.

The repository has GitHub immutable releases enabled. Once the draft release is
published, GitHub locks every attached asset and the associated tag against
modification or deletion and creates a release attestation. This ensures a pack
referenced by a published replay remains available at its immutable URL. If a
future archival migration is needed, publish and verify the archival replacement
before retiring the repository or release; never replace an existing pack/tag.

## Release workflow

The [Lexicon release workflow](../.github/workflows/lexicon-release.yml) can run
manually as a non-publishing reproducibility check. A matching `lexicons-v*` tag
also publishes after every check succeeds.

The workflow:

1. builds both packs and all corresponding materials twice from the pinned
   upstream archives;
2. compares the complete filename sets and every byte;
3. verifies compiled archive sizes/SHA-256 values against `registry.toml`;
4. verifies upstream source sizes/SHA-256 values against `sources.toml`;
5. verifies each compiled manifest identity, source, and license;
6. decompresses the legible key assets and requires their line counts to match
   the pack manifests, while requiring nonempty row-level audits;
7. creates a deterministic build-materials archive and `SHA256SUMS` covering
   every release attachment;
8. uploads all assets to a draft, then publishes it under repository release
   immutability and verifies the resulting attestation.

The complete attachment set includes:

- compiled English and French `.tar.gz` packs;
- exact pinned SCOWLv1 and Morphalou source archives;
- deterministic curated `keys.txt.gz` and source-row `audit.jsonl.gz` files;
- exact source registry, pack registry, licenses, notices, curation policy,
  pack/build instructions, and release metadata;
- a deterministic archive of the Rust builder, lockfile, policies, curation,
  licenses, workflows, and documentation;
- release notes and `SHA256SUMS`.

## Local rehearsal

Build twice into fresh directories outside Git and compare them:

```bash
cargo xtask lexicon build --from-source --release-materials --output /tmp/release-a
cargo xtask lexicon build --from-source --release-materials --output /tmp/release-b
diff <(find /tmp/release-a -type f -print | sort) \
  <(find /tmp/release-b -type f -print | sort)
```

Then package one verified build:

```bash
cargo xtask lexicon release-package \
  --input /tmp/release-a \
  --output /tmp/release-assets
(cd /tmp/release-assets && sha256sum --check SHA256SUMS)
```

Do not create or push the release tag until the release configuration, registry,
ruleset pins, and generated byte identities are final on `main`. The workflow
publishes only when the pushed tag exactly matches `release.toml`.
